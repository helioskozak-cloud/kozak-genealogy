# Generational Tree Plan — a Third Graph View Mode

**Status at time of writing:** `main` @ `b7986bb`. This plan does not modify `index.html`, `data/family.json`, or anything else — it is the design + schedule for a future set of sessions. It builds on, and does not replace, `PRESENTATION_REDESIGN_PLAN.md` (unified Graph tab, Branch + Chronological view modes) and `SITE_REWORK_PLAN.md` (data-driven Sources/Open Questions/Business tabs, `jumpToPerson`/`jumpToSource`/`jumpToBusiness`/`jumpToPart` cross-links). The Pedigree tab remains untouched and out of scope, as it has been in both prior plans.

## 1. The problem, in the owner's words

> "It's still really unclear from this graph who belongs to each generation, i.e. who are the children and who are the spouses."

Neither shipped Graph view mode encodes generation as a readable visual axis:

- **Branch view** (force-directed) distinguishes parent/child (solid gray line) from spouse (dashed purple line) only via a small legend at the top of the tab — not scannable across ~100 nodes in a physics-settled cluster where a spouse can end up above, below, or beside their partner depending on how the simulation happened to settle.
- **Chronological view** sorts by raw birth year on a single timeline axis. This does *not* group by generation: a grandparent and grandchild who happen to be ~20-25 years apart in age can land near each other on the timeline, while true parent-child pairs can be born decades apart and end up far apart on the same axis. Birth year and generation are correlated but not the same thing, and the mismatch is exactly what makes this view confusing for "who belongs to which generation."

The old Pedigree tab (still live, untouched) solved this with clear top-to-bottom-labeled generational tiers ("Focal Person / Parents / Grandparents / Great-Grandparents…") in four separate ahnentafel tables. That clarity is real, but it doesn't transfer directly: Pedigree's math (`genOf`/`rowOf`, `index.html:1209-1215`) is a per-tree ahnentafel formula (father = 2k, mother = 2k+1) that only works for four disconnected strict binary trees rooted at four different focal people — not the 100-person unified graph the Graph tab now renders from `data/family.json`.

**Goal:** a third Graph-tab view mode, "Generational view," that puts generation on the vertical axis the way Pedigree's tables did, computed correctly across the *entire* unified graph (not four separate trees), while keeping the interactivity Branch/Chronological already have — hover, click-to-panel, drag, zoom/pan.

## 4. Resolved design

### (a) How is generation computed for every one of the 100 people?

**Not** `genOf`/ahnentafel math (wrong tool — see §3, four incomparable per-tree numberings). **Not** a naive per-node longest-path either — §3 shows that alone produces 9 wrong answers for married couples, including one off by 3 generations. The resolved algorithm, run once client-side after `data/family.json` loads (same place the existing `byTree`/`EXTRA_EDGES`/`addEdge` code already runs, `index.html:2707-2766` — zero changes needed there, this is new code that consumes the same `nodes`/`edges` arrays that code already produces):

1. **Union-find over `spouse`-type edges only.** Every married pair is merged into one "family unit" group. (`disproven-link` and `same-person` edge types are excluded from this step — see §4e.)
2. **Condense `parent-child` edges onto groups.** For every `parent-child` edge `(p, c)`, add a condensed edge `group(p) → group(c)`, skipping any that would be a self-loop.
3. **Cycle check on the condensed graph.** A cycle here means a family unit is its own ancestor through some path — the concrete failure mode a real cousin-marriage (or a data error) would produce. If found, break it by un-merging the specific spousal pair whose union created the cycle (that pair reverts to each keeping their own independently-computed generation) and log it plainly in the existing `#graph-status` line (the same place `orphans`/unconnected counts already surface today) so a real cousin-marriage case is visible and reviewable, not silently mishandled. On the current dataset, zero cycles are produced (verified) — this step is a safety net, not a live concern today.
4. **Longest-path layering (Kahn's-algorithm topological order)** over the condensed DAG: a group with no incoming condensed edge is generation 0; every other group's generation is `1 + max(parent group's generation)`. This is the same `layeringLongestPath` strategy d3-dag offers (§2), applied to condensed family-unit nodes instead of raw persons.
5. Every person inherits their group's generation number. A person in a group that has **no parent-child-derived generation at all** (the 37 fully-disconnected orphans from §3, plus any spouse-only group with no ancestry on either side) gets an explicit `generation: null` — not a guess, not forced into row 0. This directly answers the "genuinely ambiguous" question in the task brief: ambiguity isn't resolved by picking a number, it's resolved by an honest "unassigned" bucket, laid out separately (§4d).

This directly fixes all 9 mismatches in §3's table: Peter George McGivney and Helen R. Relihan become one family unit, and that unit's generation is `1 + max(both of their parent-condensed-edges)` — the deeper (more-descended) of the two independently-known chains, which is the only choice that doesn't contradict either side's own parent-child ancestry.

**One accepted limitation, stated honestly rather than glossed over:** longest-path-from-root numbering is *relative to each root chain's own documentation depth*, not to a shared calendar timeline. Two root ancestors from *different, never-parent-child-connected* chains (e.g. the McGivney-Ireland chain rooted at Nicholas McGivney b.~1760, versus the `mcn` tree's own root, which has no parent-child bridge to any other chain — only a `disproven-link`) will both be labeled "generation 0" even though they weren't contemporaries. This is a real, inherent limit of layer-assignment-by-ancestry-depth (not a bug introduced by this plan) and it is why every genealogy tool in §2's reference set treats generation as a *relative* structural axis, not a literal century marker — the row labels in §4d are worded accordingly ("Generation 0 (earliest documented ancestor in this line)", not a year range) to avoid implying false precision.

### (b) Layout algorithm

**Y-axis = generation, one fixed-height row per generation number** (row 0 at the top — earliest known ancestors — descending toward the present, matching the drop-line-chart convention in §2 and the "top-to-bottom" clarity the owner praised in the old Pedigree tables). This is the "otherwise unambiguous axis" the task brief asks for: unlike Chronological view's birth-year axis, two people in the same generation row are *structurally* siblings/cousins/in-laws of that tier, not just coincidentally close in age.

**X-axis within a row = crossing-minimized ordering**, a lightweight two-pass version of Sugiyama's crossing-minimization phase:
- Seed each row's initial left-to-right order by birth year (reusing the existing `birthYearOf()` helper, `index.html:2128-2135`, already used by Chronological view) — deterministic and roughly age-ordered before any crossing-reduction runs.
- Run 2–4 barycenter sweeps (alternating top-down and bottom-up passes, the standard simplified Sugiyama crossing-reduction heuristic — the same family of technique as d3-dag's `decrossTwoLayer`): reorder each row by the mean X-position of each node's connected neighbors in the adjacent row, repeat until stable or the pass cap is hit. This is a well-understood, cheap (`O(rows × nodes × avg-degree)`) heuristic, not an exact crossing-minimizer (exact crossing minimization is NP-hard, which is why every real implementation — d3-dag included — uses a heuristic here too).
- **Spousal adjacency:** within a family unit's shared row, place the spouse pair next to each other and center their children's row-position under the pair's midpoint (the exact drop-line convention from §2: "the marriage line... the children's connecting vertical line descends from the middle of the marriage line").

**A structural simplification this problem gets "for free" that general Sugiyama doesn't:** because generation is defined as `1 + max(parent generation)` (§4a), every `parent-child` edge spans *exactly* one row by construction — there is no case of a parent-child edge skipping a generation. General Sugiyama needs dummy/virtual nodes to route edges that span multiple layers; this layout never needs them for `parent-child` edges. `spouse` edges are always within-row by construction (§4a merges spouses into one group/row). Only `disproven-link` edges can span an arbitrary row gap (§4e) — and there are exactly 2 of them today, both 1 row apart, rendered as a plain diagonal line with no dummy-node routing needed.

**Not adopted:** the full 5-phase Sugiyama framework or the `d3-dag` library as a dependency. Layer assignment (phase 2) and a lightweight crossing-reduction pass (phase 3) are the two phases that actually matter for readability here; cycle removal is a 6-line union-find safety net (§4a step 3, cycles aren't expected in practice); coordinate assignment (phase 4) is the barycenter X-ordering above, reusing this codebase's existing `CARD_W`/`CARD_H` grid math instead of a general coordinate-optimization pass; edge routing (phase 5) is unnecessary because no edge spans multiple layers (previous paragraph). This mirrors the same "generalize the existing hand-rolled renderer, don't adopt a new library" call the original redesign plan made about `family-chart` (§2) — same reasoning, same conclusion, applied to a different node.

### (c) Spouse row placement

**Confirmed, not overridden:** spouses share their partner's generation row, per the drop-line convention researched in §2 and implemented via the union-find step in §4a. This was verified as the *necessary* choice, not just the conventional one — §3's mismatch table shows what happens without it (a 3-generation-gap married couple). The one nuance beyond "confirmed": when a family unit merges two people with *different* independently-derived generations (e.g. Peter George McGivney's deep Irish ancestry vs. Helen R. Relihan's shallower Relihan-tree ancestry), the unit's generation is the deeper (more-descended) of the two — the only value that doesn't contradict either side's own parent-child chain (going shallower would place one of them *above* their own documented parent).

### (d) Coexistence with existing card/panel/hover components

**Reused as-is, zero changes:** `nodeClr()`/`CLR` (branch/confidence coloring), the card rect/text/badge rendering block (`index.html:2506-2525`, copied verbatim as the node template — same `CARD_W`/`CARD_H` = 92×30), `showTip`/`positionTip`/`hideTip` (hover tier), `showGraphPanel`/`renderDetailPanel` (click tier, including `sourcesHtmlFor`/`partsHtmlFor`/`openQuestionsHtmlFor`/`businessHtmlFor`), `zoomBehavior`/`fitGraph` (zoom/pan + auto-fit-to-bounds), `EDGE_STYLE` (edge coloring — parent-child solid gray, spouse dashed purple, disproven-link dashed red), `jumpToPerson` (already generic over "whichever render pass ran last," per its own comment at `index.html:2277-2285` — no changes needed there either).

**New, but small:**
- A static per-node `{x, y}` assignment (this section's §4b algorithm) — same shape of output as `renderChronoGraph` already produces (static positions, no physics), so it plugs into the existing `nodeG`/drag/click/hover wiring pattern with no new interaction code, only new coordinate math.
- Drag behavior: unlike Chronological view (which draws no edges, §`index.html:2637-2640`'s own comment explains why), Generational view *does* draw `parent-child`/`spouse`/`disproven-link` edges (drop-line clarity requires them — that's the whole point of this mode). So the drag handler needs to reposition the small number of edges touching the dragged node (a cheap array filter over 79 edges, no full-graph re-tick — simpler than Branch view's force-simulation `tick()`, since there's no physics to restart).
- Row labels + an "unassigned generation" lane, both new but directly modeled on shipped patterns: row labels reuse the Pedigree tab's `gLbls` idea (`index.html:1338`, "Focal Person/Parents/Grandparents…") adapted to numbered rows with the honest phrasing from §4a's limitation note; the unassigned lane is a grid of the 37 disconnected + any un-groupable spouse-only nodes, positioned exactly like `renderChronoGraph`'s existing "Unknown birth year" lane (`index.html:2592-2602`) — same deterministic-column/row grid code, different label ("Generation unclear — no traced parent/child link") and a legend entry, not new layout logic.
- **Vertical space per row:** *not* variable per generation despite generation sizes varying (17/14/14/14/2 today) — row height stays constant (`CARD_H` + fixed margin). What varies is row *width* (more people in a row → more horizontal space, exactly like Chronological view's timeline already grows wider for busier years), handled by the same `fitGraph` uniform zoom-to-fit every other mode already uses. No new vertical-scaling logic needed.

### (e) Disproven-link and same-person edge handling

**Disproven-link:** excluded entirely from generation computation (§4a step 1 and step 2 explicitly only consume `spouse` and `parent-child` types) — a disproven hypothesis must never pull two people into the same family unit or condense a parent-child layer between them. It is still *drawn*, exactly as styled today (`EDGE_STYLE['disproven-link']`, dashed red), connecting wherever the two nodes land after generation assignment. Checked against real data (§3): both current disproven-links span exactly 1 row, rendering as a short diagonal — not a problem today, and the design doesn't assume it stays that way (no code path requires disproven-links to be short).

**Same-person:** the `EDGE_STYLE['same-person']` entry and its legend line are dead code today — the three duplicate Thomas P. Relihan / John Relihan Sr. / Ann Duley Relihan records that would have used it were merged into single canonical `tpr-*` records in commit `f22941d` ("Merge 3 duplicate Relihan person records"), so no `same-person` edges exist in the current `EXTRA_EDGES` or ahnentafel-derived edge set. If one is ever reintroduced (a future duplicate found before it's merged), it should be treated the same as `disproven-link` for generation purposes — excluded from union-find and condensation, drawn as-is — for the same reason: it asserts identity/near-identity, not ancestry.
