# Site Rework Plan — Relationships as the Organizing Principle

**Status at time of writing:** `main` @ `295be48`. This plan does not modify `index.html`, `data/family.json`, or anything else — it is the design + schedule for a future set of sessions. It builds on, and does not replace, `PRESENTATION_REDESIGN_PLAN.md` (Nights 1–7, already shipped): the unified person Graph tab with Branch and Chronological view modes, `data/family.json` as the structured data layer, and Overview renamed to Research Log.

## 1. The problem, restated with numbers

The first redesign pass fixed the *person* view (Graph tab). It never touched four other tabs, and the owner's complaint is specific to each:

| Tab | Current state (verified against the live data/HTML, not assumed) |
|---|---|
| **Research Log** (43 Parts, formerly Overview) | Pure reverse-chronological scroll. No search box, no sort control, no filter. `parts{}` in `family.json` already has `date`, `sourceRefs[]`, `personRefs[]`, `summary` per Part — the data to support search/sort/filter exists; the UI for it does not. |
| **Missing Links** | 17 cards in `index.html` (verified: `grep -c class="mname"` in the tab range = 17). 3 of those 17 are already marked `RESOLVED` in the underlying `openQuestions[]` data but the static HTML still shows all 17 with no active/resolved split. |
| **Sources** (S01–S123) | 123 rows. Only 4/123 have anything URL-shaped in `citation` or `type` at all (checked programmatically), and even those are unrendered plain text, not `<a href>`. Worse: `sources{}` in `family.json` links to people only via free-text `person`/`relatedPerson` strings, not person IDs — a substring match against the 66 known person names hits only 31/123 (25%). The other 75% either use a name form that doesn't literally appear in a person record (nicknames, differently-ordered names) or name someone with no person record at all (e.g. "Msgr. G. Thomas Burns," "Relihan Landscape Construction Co."). There is no forward *or* backward link between a source row and a graph card today. |
| **Search Queue** | ~24-25 groups, already 100% folded into `openQuestions[]` (Night 3) with `origin: "search-queue"` (25 entries) sitting right next to `origin: "missing-links"` (22 entries) in the exact same array shape, on the exact same person records. The tab itself was never retired — it's live HTML duplicating data that has an already-unified home. |
| **Business & Property** | **Not in `family.json` at all** — confirmed by reading the schema (`meta`, `branches`, `persons`, `sources`, `parts`; no `business`/`property` key exists) and the tab's HTML (`index.html:2122-2185`), which is 100% hand-written cards with zero data-driven linkage to any person record. 5 records total (2 businesses, 3 properties). |
| **Pedigree** | Explicitly exempt — out of scope for this plan, per the owner. |

The owner's framing is the important part: *"I really want a whole re-work of the site that presents the mounds and mounds of data in a way that makes it clear to a viewer how all these points are related."* That's a request for **relationships as the organizing principle across every tab**, not just inside the Graph tab. Right now Sources, Open Questions (Missing Links + Search Queue), Business & Property, and the Research Log narrative are four islands with their own shapes, three of which (Sources, Open Questions, Business) have no reliable link back to the person graph, and one of which (Business) isn't in the data layer at all.

## 2. Reference points researched

Extending the reference set the first redesign plan used (WikiTree Compact Tree, `family-chart`, D3 collapsible-tree patterns, FamilySearch Timeline Grids, Ancestry hover-preview), five more directly answer *this* plan's question — how to connect evidence/questions/records to entities without turning the graph into a 250-node soup:

| Example | What's good about it | How it applies here |
|---|---|---|
| [Wikidata statement + reference model](https://www.wikidata.org/wiki/Help:Statements) | Every claim on an entity ("date of birth: 1891") carries its own `reference` (source + retrieved-date), attached to the *statement*, not floated as a separate node in the entity graph. Wikidata deliberately does **not** render sources as first-class nodes alongside entities on the same graph. | Direct precedent for keeping sources *off* the person graph and hanging them off individual facts instead — which is close to what `sourcesHtmlFor()` in the existing Graph tab already does (renders inline source rows inside the person panel, not as separate graph nodes). Validates extending that pattern rather than promoting sources to node status. |
| [WikiTree source citations](https://www.wikitree.com/wiki/Help:Sources) | Footnote-style citations on a profile, each one a real clickable link back out to the archive/database, with an unsourced-fact warning badge when a claim has none. | Exactly the "doesn't actually link to sources" gap — a citation should be an `<a href>` when a URL exists, and the confidence badge (already `bc/bp/bt/bx` on this site) should visibly flag facts with zero attached source. |
| [FamilySearch "attaching sources to the Family Tree"](https://www.familysearch.org/en/wiki/Attaching_Sources_to_the_Family_Tree) | Sources are attached directly to a specific person profile with a required "reason this source belongs" field — the link between source and person is a first-class, structured relation, not inferred from matching text. | Precedent for the biggest data-model fix this plan makes: `sources[].personRefs[]` as real person IDs (resolved once, stored, not re-derived by regex every page load) instead of the current free-text `person`/`relatedPerson` strings that only 25% substring-match. |
| [Zotero collections + tags](https://www.zotero.org/support/) | A reference manager where every citation is filterable/searchable/taggable across overlapping collections, not a flat, unsorted table. | Model for the Sources tab becoming filterable (by person, by confidence, by branch) instead of a 123-row static table — the same "search and sort" ask the owner made for Research Log applies just as much to Sources, which is currently just as unsearchable. |
| [Connected Papers / Litmaps-style citation graphs](https://www.connectedpapers.com/) | These tools *do* visualize evidence relationships as a graph — but as a deliberately separate, secondary graph you open from a paper, never merged into one mega-graph with every entity type at once. | Reinforces the same architectural call from the other side: even tools whose whole purpose is graphing evidence keep it as a filtered, on-demand secondary view rather than one undifferentiated node soup. Supports "don't put sources/questions/business records in the main Graph" — but *does* support a future optional secondary lens if ever wanted (explicitly out of scope here; see §3). |

## 3. The core architecture decision

**Recommendation: the Graph tab stays person-only. Every other tab becomes person-linked, bidirectionally, using and extending a navigation pattern that already exists.**

Reasoning:

1. **Scale.** 68 persons + 123 sources + 47 open questions + 5 business/property records ≈ 243 nodes if all typed as graph nodes. The existing force-directed layout is already tuned (three commits: "pin drifting orphans," "chronological layout fix," "spread-corrected") to stay readable at ~70 person-nodes. Nearly quadrupling node count on the same canvas would directly undo that tuning work and contradict the owner's own stated goal — "makes it clear... how all these points are related" is a clarity request, and a denser graph is the opposite of clearer.
2. **Every reference point above independently reaches the same answer.** Wikidata hangs references off statements, not the entity graph. WikiTree and FamilySearch attach sources to profiles as structured but subordinate data. Even citation-graph tools that specialize in visualizing evidence relationships keep that as a separate view, not a merged one. There is no example in the reference set of a production tool merging evidentiary/administrative record types into the same visual graph as its core entities.
3. **The forward-link mechanism the owner wants already exists in miniature** — it just runs one direction. `showGraphPanel()` in the Graph tab (`index.html:2334-2340`) already calls `sourcesHtmlFor(node)` (renders each linked source's type/citation/date/confidence inline in the person panel) and `partsHtmlFor(node)` (renders clickable "Part N →" links that call `jumpToPart(n)` to switch tabs and scroll to the right Research Log block, `index.html:2306-2332`). This plan's job is (a) making that forward link *reliable* (today it's regex-recovered from free text, not a stored relation) and (b) building the **reverse** link — from a Sources row, an Open Question card, or a Business record back to the person's graph card — via a new `jumpToPerson(id)` helper that generalizes the existing `jumpToPart(n)` pattern exactly.

So the answer to the plan's core question is: **not** new node/edge types in the person graph; **yes** to making every table/card view carry a real `personRefs[]` relation and a working link in both directions. Relationships remain the organizing principle — they just fan out from the person graph as the hub, rather than living inside it as additional node types.

## 4. Resolved data model changes to `family.json`

`schemaVersion` bumps from 2 to 3. All changes are additive to existing keys except the new `businesses[]` collection — nothing existing is renamed or removed, so the whole existing Graph tab renderer keeps working unmodified against the new file.

```jsonc
{
  "sources": {
    "S01": {
      // existing fields unchanged: num, person, relatedPerson, clue, type, citation, date,
      // confidenceClass, confidence, notes, dataQuirk
      "personRefs": ["thomas-p-relihan-sr", "..."],   // NEW — resolved person IDs, not re-derived from text
      "url": null                                      // NEW — nullable; populated only where a real online
                                                         // record exists (FamilySearch/Ancestry/WikiTree page,
                                                         // data.ct.gov query, archive.org scan, etc.)
    }
  },
  "businesses": {                                       // NEW top-level collection
    "biz-relihan-landscape": {
      "kind": "business",                               // "business" | "property"
      "name": "Relihan Landscape Construction Co., Inc.",
      "status": "confirmed",                             // reuses the bc/bp/bt/bx vocabulary
      "personRefs": ["thomas-p-relihan-jr", "carla-guarino-relihan"],
      "sourceRefs": ["S11"],
      "facts": [["Principal address", "924 Baldwin Rd, Woodbridge CT"], ["Incorporated", "December 7, 1988"]],
      "leadText": "Deed search 924 Baldwin Rd..."          // the existing "→ next step" callout, kept verbatim
    }
    // ... 4 more records (Relihan & Irwin, 3 Baldwin Rd parcels)
  }
  // persons{}, parts{}, and openQuestions[] (already nested per-person) are UNCHANGED in shape.
}
```

Specifically **not** changed, with reasoning:

- **`openQuestions[]` stays nested under each person** (not promoted to a new top-level `openQuestions{}` collection). It is already the single source of truth merging Missing Links + Search Queue (Night 3) — duplicating it into a second top-level index would create exactly the two-places-for-one-fact problem this whole plan exists to remove. The new unified Open Questions tab (§5) flattens it into a list client-side at render time, the same way the Graph tab already reads `persons{}` client-side. `origin` (`missing-links` / `search-queue`) is **kept** on each entry as provenance metadata but is **not** surfaced as a UI split — the owner explicitly called that distinction meaningless duplication; `status` (`open`/`resolved`) is the UI split that matters.
- **`persons{}` does not gain a `relations{}` object** in this pass (the original plan's §4 schema sketch proposed one; Nights 1-3 shipped without it, using the original ahnentafel `sourceKey`/`childGroupInfo` shape plus a hand-maintained `EXTRA_EDGES` array in the Graph tab script instead). That gap is real but is a Graph-tab-internal concern already flagged in the code's own comments (`index.html:2366-2372`) — out of scope for *this* plan, which is about connecting the other four tabs to the graph, not re-deriving the graph's own edge model. Not touching it also means zero risk to the already-tuned Graph tab.
- **`sources[].url` is nullable, not required.** Most citations (funeral home records, microfilm, parish registers) have no online equivalent and forcing one would mean fabricating links — exactly the kind of fake-looking citation the owner is trying to get away from. A null `url` renders as a plain citation with a small "no online record" indicator instead of a broken or fake link.

## 5. Resolved UI / navigation design

- **New shared helper `jumpToPerson(id)`** (Graph tab script), the direct generalization of the existing `jumpToPart(n)`: switches to the Graph tab, finds the node by ID, centers/pans to it, and calls the existing `showGraphPanel()` to open its detail panel. Every new cross-link in §5 below is a call to this one function — no new navigation mechanism, one function reused everywhere, matching how `jumpToPart` is already reused across the Research Log-linking code.
- **Person detail panel** (`gpanel`, driven by `showGraphPanel()`) gains two new sections, generalizing the existing `sourcesHtmlFor()`/`partsHtmlFor()` pattern:
  - `openQuestionsHtmlFor(node)` — any `openQuestions[]` entries on that person, status badge (open/resolved), linking to the Open Questions tab entry.
  - `businessHtmlFor(node)` — any `businesses{}` records whose `personRefs` includes this person.
- **Missing Links + Search Queue tabs are retired and merged into one new "Open Questions" tab**, built from a client-side flatten of `persons[*].openQuestions[]` (same load-time pattern the Graph tab already uses for `family.json`): an **Active** / **Resolved** toggle (not active/missing-links/search-queue — see §4), sortable by `priority`, each card linking forward to its `sourceRefs` (Sources tab) and back to its person (`jumpToPerson`).
- **Sources tab** becomes a filterable table (by person, confidence, branch — Zotero-style faceting) with `citation` rendered as `<a href="{url}">` when `url` is present, a "◂ View in Graph" link per row calling `jumpToPerson` off the new `personRefs[]`, and a small "no online record" label when `url` is null rather than leaving the cell looking broken.
- **Business & Property tab** is rebuilt to render from the new `businesses{}` collection instead of hand-written HTML, with the same person-link and source-link treatment as Sources.
- **Research Log** gains a search box (full-text across `title` + `summary`, client-side — 46 parts is small enough that no server/index is needed), a person filter (typeahead over the same 68 person names, matching `parts[*].personRefs`, reusing data already populated Night 2-3), and a sort toggle: default stays **Part number, descending** (unchanged, preserves the existing bookmark/mental-model), with a second option for **session date, ascending** (chronological) — using the `date` field already on every part. Both are pure re-orderings of the same DOM blocks; no new data needed.
- **Graph tab, both view modes, and Pedigree are untouched** — this plan adds two rendering functions to the panel and one navigation helper; it does not touch the force-simulation, the chronological-layout math, or `EXTRA_EDGES`.

## 6. Answers to the five specific questions

**(a) Data model changes** — see §4: `sources[].personRefs[]` + `sources[].url`, new `businesses{}` collection. `openQuestions[]` and `parts{}` need no schema change, only a new UI reading them.

**(b) Search Queue disposition** — **delete the tab outright**, do not keep it even as a filtered view. `openQuestions[]` already fully absorbed it (Night 3); the tab is pure duplication today. Missing Links is renamed **"Open Questions"** (broader than its old name, since it now equally covers what used to be Search Queue) and gains the active/resolved split the owner asked for.

**(c) Research Log search/sort, concretely** — full-text search across all Parts' `title`+`summary` (not the entire original prose blocks — those aren't separately captured in `family.json`, only the mined summary is; searching the summary is sufficient for "find the part about X" and doesn't require re-extracting 20k words of prose a second time), **plus** filter by person (via existing `partRefs`), **plus** a sort toggle defaulting to current Part-number-descending order with a chronological-by-`date` alternative. All three, not a choice between them — they answer three different questions ("what mentions X," "what's newest," "what happened in what order") and the data to support all three already exists in `parts{}`.

**(d) Sources: hyperlinks only, or deeper restructure** — **deeper restructure.** Real hyperlinks alone would still leave a 123-row table with no connection to the person graph, which is the complaint the owner separately named ("doesn't... have any connection back to the people or graph"). Both `personRefs[]`-driven cross-links and a filter/search UI are needed to actually resolve the complaint, not just the literal word "link."

**(e) Business & Property realism** — **extract into `family.json` now.** It is small: 5 records total (2 business cards, 3 property parcels), all already broken into clean HTML `<table>` rows with labeled fields — structurally closer to the original Sources-table extraction (mechanical, low-risk) than to the 43-Part prose-mining effort. There is no honest argument for leaving 5 records as a hand-maintained appendix while every other tab becomes data-driven; the extraction cost here is a fraction of a session, not a multi-session undertaking.

## 8. How this avoids losing or breaking prior work

- **Nothing in `persons{}`, `parts{}`, `branches[]`, or the Graph tab's rendering/simulation code is modified.** Every data-model change in §4 is a new field or a new top-level collection; every UI change is a new tab-rebuild or a new panel section, not an edit to `setGraphMode`, the force layout, the chronological-layout math, or `EXTRA_EDGES`.
- **`openQuestions[]` (Night 3's merge of Missing Links + Search Queue) is read, not re-derived or re-merged** — this plan trusts that data and only builds the UI Night 3 never got around to building for it, per the owner's own observation that the tabs were "never retired... to reflect that."
- **The Night 6 cutover** (Graph as default tab, Research Log renamed, `data-part` attributes for `jumpToPart`) is the foundation `jumpToPerson` and the new search/filter UI are built on top of, not around — Session 5 explicitly reuses the existing `data-part` targeting rather than inventing a second addressing scheme.
- **Session boundaries are additive-only by construction**: Session 1 only touches `family.json`; Sessions 2-5 each touch exactly one tab's HTML/script plus the shared `jumpToPerson` helper (added once, in Session 2, reused after); Session 6 touches nothing except verification and any drift fixes. A user stopping after any session has a strictly-better, fully-working site — no session leaves the live site in a half-migrated state, matching the "no half-finished implementations" standard the prior plan (and this one) both hold to.
- **Confidence badge vocabulary (`bc`/`bp`/`bt`/`bx`/`br`/`be`) is reused everywhere**, never redefined — the new Open Questions and Business & Property tabs use the exact same badge classes and colors already established for Sources and the Graph tab, so the visual language stays one system across all six tabs, per the owner's actual ask: one coherent way to see how everything relates, not a seventh new UI convention.
